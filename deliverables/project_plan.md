## 3. 开发阶段划分

### 第一阶段：数据层改造（T1-T5）

```
目标：数据库表结构调整，Repository 层适配新业务
交付：
  - 新 materials 表（code/name/version）
  - 新 processes 表（material_code/process_name/unit_price）
  - 移除 worker_processes 表
  - MaterialRepository 新增 search() 方法
  - ProcessRepository 新增 get_process_names()，移除工人分配方法
  - WorkerRepository 新增班组管理方法
  - 电气行业种子数据
```

### 第二阶段：服务层适配（T6-T7）

```
目标：Service 层适配 Repository 的新接口
交付：
  - WorkerService 新增班组管理
  - ProcessService 移除工人分配
```

### 第三阶段：UI 层重构（T8-T12）

```
目标：三个对话框按新设计重构
交付：
  - MaterialDialog：编号/名称/版本三列
  - WorkerDialog：树形结构（班组→工人），班组下拉选择
  - ProcessDialog：去掉工人分配，物料/工序改为可搜索下拉
  - dashboard_view：工序显示格式更新
```

### 第四阶段：测试与验证（T13-T14）

```
目标：确保所有变更正常工作
交付：
  - 物料 CRUD 正常（含多版本）
  - 工人树形展开折叠正常
  - 班组管理（添加/删除班组）正常
  - 工序物料搜索过滤正常
  - 主界面工序选择正常
  - 旧功能无回归
```

## 5. 风险预判与应对措施

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| 表结构变更导致旧数据库不兼容 | 高 | 中 | 使用版本检测，自动重建数据库；提示用户备份 |
| 树形结构交互复杂（Tkinter Treeview） | 中 | 中 | 参考官方文档实现父子节点操作 |
| Combobox 搜索过滤响应慢 | 低 | 低 | 物料数据量小（<200条），客户端过滤即可 |
| 删除班组误操作 | 中 | 高 | 二次确认对话框，提示将同时删除组内所有工人 |
| 工序关联物料被删除 | 中 | 中 | 删除物料时检查关联工序，提示用户确认 |
| 主界面工序格式变更影响统计图表 | 低 | 中 | 图表层只使用 process_id，显示格式变化不影响数据 |
